Synthèse Grafana

Couche de visualisation par-dessus Prometheus, pour le projet DevOps Portfolio MERN. Data source, dashboards, panneaux, options de visualisation, et industrialisation en « dashboards as code ».

1. Rôle de Grafana

Grafana ne stocke aucune métrique. C'est un outil de visualisation qui se connecte à des sources de données (ici Prometheus), les interroge en PromQL, et affiche le résultat sous forme de tableaux de bord composés de panneaux.

   Navigateur (toi)
        |
        v
   +-----------+      requete PromQL (proxy)      +-------------+
   |  GRAFANA  | -------------------------------> | PROMETHEUS  |
   | dashboards|  <-------------------------------| (data store)|
   +-----------+      series temporelles          +-------------+

2. Concepts clés

ConceptDéfinition
Data sourceLa source que Grafana interroge (Prometheus, Loki…). Mode proxy = Grafana l'appelle côté serveur.
DashboardUne page composée de panneaux. Techniquement, un simple document JSON.
Panel (panneau)Une visualisation d'une ou plusieurs requêtes PromQL.
Type de visualisationLa forme du panneau : Time series (courbe), Gauge (cadran), Stat (gros chiffre), Table…
Variable de dashboardMenu déroulant en haut du dashboard (ex. job, instance) pour filtrer dynamiquement et rendre le dashboard réutilisable.
ProvisioningConfigurer Grafana « en code » au démarrage (data sources et dashboards via des fichiers), au lieu de cliquer dans l'UI.

4. Dashboards as code (la bonne pratique)

Plutôt que de créer les dashboards à la main (perdus si le pod redémarre), on les versionne en JSON dans Git et Grafana les charge automatiquement via le provider.

# Régénérer le ConfigMap des dashboards à partir des fichiers JSON
kubectl create configmap grafana-dashboards -n monitoring \
  --from-file=monitoring/grafana/dashboards \
  --dry-run=client -o yaml | kubectl apply -f -

kubectl rollout restart deployment/grafana -n monitoring

Importer un dashboard communautaire (ex. « Node Exporter Full », ID 1860) : UI → Dashboards → New → Import → saisir l'ID. Pour le versionner ensuite, déposer son JSON dans le dossier dashboards/ et régénérer le ConfigMap.

5. Commandes utiles

# Accéder à l'UI Grafana (cluster kind => port-forward)
kubectl port-forward -n monitoring svc/grafana 3000:3000
# -> http://localhost:3000   (admin / admin)

# Réinitialiser le mot de passe admin (pas d'email de reset : SMTP non configuré pour Grafana)
kubectl exec -n monitoring deploy/grafana -- grafana cli admin reset-admin-password admin

6. Pièges rencontrés & bonnes pratiques

PiègeLeçon
Pas de sélecteur de data source à l'importNormal si une seule source par défaut : Grafana la lie automatiquement.
Pod Grafana bloqué en PendingCluster mono-node à court de CPU + stratégie RollingUpdate (besoin du double). Passer en strategy: Recreate.
Mot de passe oublié, pas d'emailGrafana n'a pas de SMTP (seul Alertmanager en a). Reset via grafana cli.
Mot de passe / dashboards perdus au redémarrageStockage éphémère. Le mot de passe redevient celui de GF_SECURITY_ADMIN_PASSWORD. Pour persister : un PVC.
Dashboard provisionné « datasource not found »Fixer un UID stable sur la data source (uid: prometheus) et le référencer dans le JSON.
Identifiants admin en clairOK en local ; en prod, passer par un Secret Kubernetes.

8. Prochaines étapes

  • Plus de dashboards as code : versionner le 1860 et d'autres dashboards métier.
